    Spillover categories

    Use these definitions to choose the closest category when you fill in spillover on the Spillover tab. Pick one primary category per issue; use the reason field for specifics.

    Category Definition Sub-types / examples
    Defect Blocker

    Items delayed because open bugs or defects needed investigation or resolution before the team could proceed.

    		—
    Dependency

    Tickets blocked by dependencies — internal (team availability, cross-team work) or external (stakeholders, vendors such as Salesforce).

    • Dependency / external blockers — reliance on other teams, tickets, or inputs outside the team's control.
    • Business / stakeholder delays — pending clarification, approvals, or responses from business stakeholders.
    • Scoped tickets — scoped for part of the functionality or QA-only validation, often depending on related work finishing first.
    • Late QA handoff — dev delivered to QA too close to sprint end, leaving insufficient test time.
    • Dependencies — blocked by other in-flight tickets or work that had to complete first.
    Scope Creep

    Requirements changed mid-sprint so the ticket could not be finished within the original sprint boundaries.

    • Requirement change — scope altered mid-sprint; rework or re-planning.
    • Analysis — still under exploration; not yet ready for development.
    • Mid-sprint assigned — story added after sprint start, reducing time to complete.
    • Reprioritised spillover — deprioritised in favour of higher-priority work.
    • Product limitation — technical product constraints required workarounds or investigation, adding unplanned effort.
    Owner Specific

    Ownership or expertise sat outside the usual G10X assignee pattern, creating a bottleneck around availability or domain knowledge.

    		—
    Estimation

    Work was under-estimated (effort or complexity), so it did not finish in the sprint — including hidden dependencies, scope expansion during development, or ad-hoc / operational work added mid-sprint.

    Dev/QA spillover — dev or QA work was in progress but could not be completed within the sprint window.

    Carryover Work

    Items expected to carry over, including spikes and other planned carryover.

    Bug defect categories

    Use this reference when you choose Category on the Bugs tab. Pick the closest match; use RCA for specifics.

    Category Definition Examples / notes
    Requirements Issues

    Defects rooted in how requirements were written, communicated, or understood.

    • Ambiguous Requirements — unclear, incomplete, or not well-defined.
    • Changing Requirements — frequent changes not properly communicated or managed.
    • Misunderstood Requirements — incorrect interpretation by dev or QA.
    Design Problems

    Issues stemming from UX or design decisions.
    • UX related issues
    Implementation Errors

    Problems introduced during design implementation: architecture, code quality, testing discipline, or operational setup.

    • Architecture — flawed system architecture.
    • Complexity — designs hard to implement or test.
    • Coding Errors — bugs in coding (including regression defects).
    • Inadequate Code Reviews — insufficient peer review.
    • Inadequate or Missing Unit Tests — lack of basic unit testing.
    • Technical Debt — quick fixes and workarounds degrading quality.
    • New Configuations — setup needed per environment.
    Testing Deficiencies

    Gaps in validation: coverage, design of tests, manual mistakes, automation, or specialized test types.

    • Inadequate Test Coverage; Poor Test Design; Manual Testing Errors
    • Automation Errors; Load Testing; E2E Testing; UAT Testing
    Process, Tools and Environment

    Workflow, tooling, documentation, environments, and configuration management.

    • Lack of Process Adherence; Inefficient Processes; Inadequate Documentation
    • Inadequate Tools; Environment Issues (dev vs test vs prod)
    • Configuration Management — configs, certificates, dependencies.
    Human Factors

    People and organizational constraints affecting quality.
    • Skill Gaps; Communication Issues; Resource Constraints
    External Factors

    Issues originating outside the core codebase.
    • Third-Party Components — integrated tools (e.g. MAO, Mulesoft, Aurus).
    Data Issues

    Problems with data correctness, completeness, duplication, freshness, or integration from source systems.

    • Data Quality Issues — incorrect, incomplete, duplicate, or outdated data (e.g. product data on Business Manager).
    • Data Integration Issues — mapping from source systems.
    Acknowledged

    QA has reviewed the ticket and is actively investigating root cause. Temporary status until a defect category is assigned.
